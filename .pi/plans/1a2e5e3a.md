# Investigation: eGuard Agent High CPU/RAM on Fedora Event Floods

## User Request
Identify likely causes of high CPU/RAM behavior during event floods on Fedora and suggest smallest high-impact fix with file-level evidence.

## Codebase Structure
- Rust workspace (13 crates)
- Linux-specific event handling: `crates/platform-linux/src/`
- eBPF ring buffer + event enrichment + detection pipeline
- Event flow: eBPF → ring buffer → enrichment → detection → telemetry

## Key Files Examined
1. `crates/platform-linux/src/ebpf/libbpf_backend.rs` - Ring buffer + record pool
2. `crates/platform-linux/src/lib.rs` - Event enrichment with caching
3. `crates/agent-core/src/lifecycle/telemetry_pipeline.rs` - Event polling & backlog
4. `crates/agent-core/src/lifecycle/runtime.rs` - Main runtime struct

## Findings

### PRIMARY HOTSPOT: Unbounded Record Sink (CRITICAL)
**File**: `crates/platform-linux/src/ebpf/libbpf_backend.rs`
**Lines**: 506-514 (push_raw_record), 583-587 (drain_record_sink)

```rust
type RecordSink = Arc<Mutex<Vec<Vec<u8>>>>;  // UNBOUNDED!

fn push_raw_record(raw: &[u8], records_sink: &RecordSink, pool_sink: &RecordPool) {
    // ...
    if let Ok(mut guard) = records_sink.lock() {
        guard.push(record);  // <-- NO SIZE CHECK
    }
}

fn drain_record_sink(records: &RecordSink) -> Result<Vec<Vec<u8>>> {
    let mut guard = records.lock().map_err(...)?;
    Ok(std::mem::take(&mut *guard))
}
```

**The Problem**:
- eBPF callbacks run in kernel context and invoke `push_raw_record` atomically
- Each callback pushes raw bytes to `records_sink` without capacity checks
- Under a flood (thousands of events/sec), the Vec can grow unbounded before being drained
- A single `poll_once()` call can accumulate tens/hundreds of MB in `records_sink`
- Each raw event is typically 1-5 KB (contains process info, file paths, cmdline, env vars from eBPF)
- In a sustained event flood: `records_sink.len() * avg_record_size = potential GBs of memory`

**Evidence**:
- No bounded buffer or backpressure mechanism exists
- `MAX_RECORD_POOL_ENTRIES` (4,096) caps only the object pool for reuse, NOT the sink itself
- Ring buffer itself has kernel-level capacity, but userspace `records_sink` has none
- `drain_record_sink` is called once per `tick()`, but between ticks a flood can accumulate massive quantities

### SECONDARY ISSUE: Enrichment Cache Misses Under Load
**File**: `crates/platform-linux/src/lib.rs`
**Lines**: 215-260 (EnrichmentCache), 354-500+ (enrich_event_with_cache)

**The Problem**:
- Process cache: 500 default entries (LRU), new process creations evict old ones
- During a flood with many unique processes (e.g., from a script exploit), cache thrashing occurs
- Each cache miss triggers expensive `/proc/<pid>/exe` and `/proc/<pid>/cmdline` reads
- File hashing: `compute_sha256_file()` (crypto-accel) can take milliseconds per file
- Container detection: calls `detect_container()` and `detect_privileged_container()` (cgroup parsing)

**Impact under flood**:
- High process/file churn = constant cache misses = frequent syscalls
- File hashing is CPU-bound and can block event processing
- Container detection per-event adds overhead (cgroup reads are safe but not free)

**Mitigation exists but not aggressive enough**:
- `strict_budget_mode` can disable file hashing
- Exclusions exist but default is permissive
- Hash finalize delay (1.2 sec) is meant to avoid redundant work but doesn't help under initial flood

### TERTIARY ISSUE: Per-Tick Processing Bottleneck
**File**: `crates/agent-core/src/lifecycle/telemetry_pipeline.rs`
**Lines**: 207-280 (next_raw_event)

**The Problem**:
- `next_raw_event()` polls eBPF once per tick
- Coalescing (file + event txn) happens before backlog enqueue
- If backlog is full, events are dropped (enforce_raw_event_backlog_cap)
- Default backlog cap: ~100,000 events (env var: EGUARD_RAW_EVENT_BACKLOG_CAP)
- Only ONE event is processed per tick (evaluate_tick → detection → send)

**Under a flood**:
- If eBPF produces 10,000 events in 100ms, but only 1 event is processed per tick
- Backlog quickly fills, then excess events are dropped
- But before dropping, they're accumulated in `records_sink` and then in backlog — high memory

## Root Cause Analysis

### Why Fedora Specifically?
- Fedora kernel is more aggressive with eBPF feature enablement (LSM hooks, etc.)
- More probes attached = more events through ring buffer
- Modern kernels = faster event production = faster backlog saturation
- Fedora defaults favor performance over resource conservation

### Memory Exhaustion Timeline
1. Flood starts: eBPF callbacks fire at high rate
2. eBPF callbacks push to unbounded `records_sink` (no checks)
3. `records_sink` grows: 0 → 1K → 10K → 100K+ entries
4. At poll time: `drain_record_sink` returns a Vec with 100K+ entries
5. Events are coalesced and added to `raw_event_backlog`
6. Even with backlog capping, the intermediate Vec from drain is GBs
7. Memory pressure → swap → CPU cycles to page in/out

### CPU Exhaustion Timeline
1. Backlog fills with coalesced events
2. `strict_budget_mode` activates (when backlog >= threshold)
3. Enrichment is stunted (skips file hashing) but detection still runs
4. File cache lookups become synchronous (no lazy hashing)
5. Process cache thrashing from unique PIDs
6. Repeated `/proc` reads for cache misses
7. System load increases, eBPF overhead increases, positive feedback loop

## Smallest High-Impact Fix

### Option A (RECOMMENDED): Bounded Record Sink with Drop Counter
**Target**: `crates/platform-linux/src/ebpf/libbpf_backend.rs`

Replace unbounded `Vec` with a fixed-size circular buffer. When full, drop oldest.

**Change**:
```rust
const MAX_PENDING_RECORDS: usize = 8_192;  // ~40-80 MB for 5-10KB records

fn push_raw_record(raw: &[u8], records_sink: &RecordSink, pool_sink: &RecordPool) {
    let mut record = ...;
    record.extend_from_slice(raw);
    
    if let Ok(mut guard) = records_sink.lock() {
        if guard.len() >= MAX_PENDING_RECORDS {
            // Ring buffer behavior: drop oldest
            guard.remove(0);
            // Track drop for observability
            // drop_count.fetch_add(1, Ordering::Relaxed);
        }
        guard.push(record);
    }
}
```

**Impact**:
- Prevents unbounded growth
- Memory cap: ~50-100 MB worst case (vs. unlimited)
- Drop counter provides visibility
- Kernel still drops excess via ring buffer overflow counter

**Lines**: ~510, add const at top of module, modify push_raw_record check

### Option B (Alternative): Semaphore-Based Backpressure
Use a counting semaphore to block callbacks if sink is too full. More complex, requires careful deadlock prevention.
