## 1. QUARANTINE SHA256 FAILURE (P0 — Restoration Data Loss)

### Problem Statement
File quarantine operations accept SHA256 hashes without case normalization, causing restoration failures on case-sensitive filesystems (Linux). The validation layer allows uppercase/mixed-case hashes, but the IOC matching layer normalizes to lowercase, creating a silent mismatch.

### Evidence

**Location 1: Quarantine Validation (Overly Permissive)**  
File: `crates/response/src/quarantine.rs:116-131`
```rust
fn is_valid_quarantine_id(id: &str) -> bool {
    !id.is_empty() && id.len() <= 128 && id.chars().all(|c| c.is_ascii_hexdigit() || c == ':')
    //                                               ☝️ ALLOWS BOTH a-f AND A-F (uppercase)
}
```
- Accepts `"ABCDEF0123456789..."` (uppercase)
- Accepts `"AbCdEf0123456789..."` (mixed case)
- No case normalization applied before filesystem path creation

**Location 2: Quarantine Path Creation (Uses Raw Input)**  
File: `crates/response/src/quarantine.rs:107`
```rust
let quarantine_path = quarantine_dir.join(sha256);
// ☝️ Uses input as-is: uppercase → uppercase path
```

**Location 3: SHA256 Generation (Always Lowercase)**  
File: `crates/detection/src/layer1.rs` (via crypto_accel)
```rust
pub fn check_hash(&self, hash: &str) -> Layer1Result {
    let normalized = hash.to_lowercase();  // ← IOC matching normalizes
    // ...compare against normalized set
}
```
- SHA256 hashes generated by eBPF/crypto layer are **always lowercase hex**
- But detection engine normalizes again during IOC matching

**Location 4: Response Action (Passes Hash as-Is)**  
File: `crates/agent-core/src/lifecycle/response_actions.rs:250-254`
```rust
let sha = event
    .file_hash
    .clone()
    .unwrap_or_else(|| synthetic_quarantine_id(event));
match quarantine_file(Path::new(path), &sha, &self.protected) {
    //                                        ☝️ No case normalization
}
```

### Failure Scenarios

1. **Uppercase Hash Path Creation:**
   - Event reports file_hash: `"ABCDEF0123456789ABCDEF0123456789"`
   - Quarantine path created: `/var/lib/eguard-agent/quarantine/ABCDEF0123456789ABCDEF0123456789`
   - Restoration lookup for lowercase path: **NOT FOUND** (case-sensitive filesystem)

2. **IOC Matching/Detection Normalization Bypassed:**
   - Quarantine uses SHA256 as filesystem path (never normalized)
   - IOC Layer (L1) normalizes for comparison but too late
   - Uppercase variant can pass validation but fail restoration

3. **Windows False Positive Security Risk:**
   - Windows (case-insensitive filesystem) masks the bug
   - Path works but inconsistent with IOC matching semantics
   - Restoration may retrieve wrong file if collision exists

### Test Coverage Gaps
- ✗ No test for uppercase SHA256 quarantine+restore roundtrip
- ✗ No test for mixed-case hash validation
- ✗ Test `quarantine_runtime_entrypoint_moves_file_and_reports_fields` uses lowercase-only: `"abcdef0123456789abcdef0123456789"` (safe case)

### Recommended Fix (3 Options)

**Option 1: Normalize at Quarantine Entry (Safest)**
```rust
fn quarantine_file(..., sha256: &str, ...) {
    let normalized_sha = sha256.to_lowercase();  // ← Add this
    if !is_valid_quarantine_id(&normalized_sha) { ... }
    let quarantine_path = quarantine_dir.join(&normalized_sha);
    // ...
}
```
- Single point of normalization
- Preserves all existing validation
- Minimal change, maximal compatibility

**Option 2: Normalize Validation Only**
```rust
fn is_valid_quarantine_id(id: &str) -> bool {
    let normalized = id.to_lowercase();
    !normalized.is_empty() && normalized.len() <= 128 && 
    normalized.chars().all(|c| c.is_ascii_hexdigit() || c == ':')
}
```
- Still requires entry-point normalization (not sufficient alone)

**Option 3: Reject Non-Lowercase (Strictest)**
```rust
fn is_valid_quarantine_id(id: &str) -> bool {
    !id.is_empty() && id.len() <= 128 && 
    id.chars().all(|c| c.is_ascii_lowercase() || c == ':')  // ← lowercase only
}
```
- Propagates normalization requirement upstream
- Breaks any uppercase input (might be desired for data migration)

## 2. MULTI-HOST CORRELATION CONFIDENCE WEIGHTING (P0 — Under-Classification)

### Problem Statement
Multi-host campaign correlation (AC-DET-186 through AC-DET-189) is 0% implemented. Current implementation counts hosts but ignores per-host confidence weights, missing Definite/High/Medium distinctions that should affect severity classification.

### Acceptance Criteria vs. Implementation

| AC | Requirement | Status | Gap |

| AC-DET-187 | Deduplication by (host, IOC, confidence) → retain strongest per host | ❌ NOT IMPLEMENTED | Only stores last_seen_unix per agent |

### Recommended Fix (5-Phase Implementation)

**Phase 1: Update Store Structure**
```go
type ConfidenceRecord struct {
    ConfidenceLevel string
    FirstSeenUnix   int64
    EventCount      int64
}

type Store struct {
    // iocSightings: IOC → (AgentID → ConfidenceRecord)
    iocSightings map[string]map[string]ConfidenceRecord
}
```

**Phase 2: Implement Confidence-Aware Deduplication**
```go
func (s *Server) ingestIocSignal(agentID, iocValue, confidence string, firstSeen int64) {
    agents, ok := s.store.iocSightings[iocValue]
    if !ok {
        agents = make(map[string]ConfidenceRecord)
        s.store.iocSightings[iocValue] = agents
    }
    
    confidenceWeight := confidenceToWeight(confidence)
    
    if existing, ok := agents[agentID]; ok {
        // Retain highest confidence for this agent
        if confidenceWeight > confidenceToWeight(existing.ConfidenceLevel) {
            agents[agentID] = ConfidenceRecord{
                ConfidenceLevel: confidence,
                FirstSeenUnix:   existing.FirstSeenUnix,  // preserve earliest
                EventCount:      existing.EventCount + 1,
            }
        }
    } else {
        agents[agentID] = ConfidenceRecord{
            ConfidenceLevel: confidence,
            FirstSeenUnix:   firstSeen,
            EventCount:      1,
        }
    }
}

func confidenceToWeight(confidence string) int {
    switch strings.ToLower(confidence) {
    case "definite": return 5
    case "veryhigh": return 4
    case "high": return 3
    case "medium": return 2
    case "low": return 1
    default: return 0
    }
}
```

**Phase 3: Implement Weighted Score Calculation**
```go
func (s *Server) computeWeightedScore(agents map[string]ConfidenceRecord) float64 {
    score := 0.0
    for _, record := range agents {
        score += float64(confidenceToWeight(record.ConfidenceLevel))
    }
    return score
}
```

**Phase 4: Implement AC-DET-188 Severity Tiering**
```go
func (s *Server) determineSeverity(hostCount int, weightedScore float64) string {
    if hostCount >= 8 || weightedScore >= 30 {
        return "outbreak"
    }
    if hostCount >= 5 || weightedScore >= 18 {
        return "elevated"
    }
    if hostCount >= 3 {
        return "advisory"
    }
    return "advisory"  // default for 3+ hosts
}
```

**Phase 5: Implement AC-DET-189 Deterministic Sort**
```go
type CampaignAlert struct {
    IocValue      string
    IocType       string
    HostCount     int
    WeightedScore float64
    Severity      string
    FirstSeenUnix int64
    AffectedAgents []string  // sorted lexicographically
}

sort.Slice(campaigns, func(i, j int) bool {
    // 1. Severity DESC (outbreak > elevated > advisory)
    severityOrder := map[string]int{"outbreak": 2, "elevated": 1, "advisory": 0}
    if severityOrder[campaigns[i].Severity] != severityOrder[campaigns[j].Severity] {
        return severityOrder[campaigns[i].Severity] > severityOrder[campaigns[j].Severity]
    }
    // 2. WeightedScore DESC
    if campaigns[i].WeightedScore != campaigns[j].WeightedScore {
        return campaigns[i].WeightedScore > campaigns[j].WeightedScore
    }
    // 3. IOC ASC
    return campaigns[i].IocValue < campaigns[j].IocValue
})
```
